## 第23篇：15｜防微杜渐：Hooks 事件驱动自动化

**核心观点：** Hooks 是 Claude Code 三大扩展机制中唯一能拦截和修改 Claude 行为的机制——它是 AI 助手的中间件，在工具调用前后插入自定义检查，把安全防线、质量守卫、审计日志从"事后检查"变成"自动运行"。

**关键概念：**
- Hooks 本质：AI 时代的中间件，针对 AI Agent 的工具调用而非 HTTP 请求
- 17 种 Hook 事件：控制点（PreToolUse/UserPromptSubmit/Stop，能阻止）、接管点（PermissionRequest，替代默认行为）、观察点（PostToolUse/SessionStart 等，只能记录）
- 三种控制谱系：Commands（任务指令）、Skills（领域知识）、Hooks（安全制度和质量规范）
- 配置位置：用户级（个人习惯）→ 项目级（团队约定，提交 git）→ 本地覆盖（临时调试）→ 子代理 frontmatter（子代理专属）

**对 news1 项目的启发：**
- news1 已有 Hooks 配置（settings.json），可以进一步完善：PreToolUse 阻止危险命令（如 `rm -rf`），PostToolUse 在文件修改后自动运行 lint
- Stop Hook 可以在 Claude 完成任务后自动运行 `pytest tests/ -v`，测试不通过就不让停

## 第24篇：16｜未雨绸缪：Hooks 高级模式与工程实践

**核心观点：** Stop Hook 是质量门控的关键——通过 `continue: true` 创建自动循环，让 Claude 不是"做完了再检查"，而是"检查通过了才算做完"；SubagentStop 事件让子代理完成时也能触发质量验收。

**关键概念：**
- Stop Hook + continue: true：Claude 认为完成 → Hook 检查 → 发现问题 → 反馈给 Claude → 继续修复 → 再次检查，直到通过
- 自动测试门控：Stop 时运行测试而非每次文件修改后，因为中间状态的测试必然失败
- SubagentStop 事件：子代理完成时触发，可以验收子代理的交付物质量
- frontmatter 内置 Hooks：子代理专属的 Hook，比全局配置更精准（如 db-reader 的 SQL 注入检查）

**对 news1 项目的启发：**
- 可以为 news1 配置 Stop Hook：每次 Claude 完成代码修改后，自动运行 `pytest tests/ -v`，失败则继续修复
- 子代理的 frontmatter Hooks 可以用于：test-runner 子代理完成后自动检查测试覆盖率，code-reviewer 完成后自动检查是否有遗漏的安全问题

## 第25篇：17｜海纳百川：MCP 协议与外部工具连接

**核心观点：** MCP（Model Context Protocol）是 AI 的 USB-C 接口，把 M×N 的适配器问题变成 M+N，让 Claude Code 从只能操作本地文件的工具进化为能连接整个数字世界的智能枢纽；已成为 Linux 基金会下的开放标准。

**关键概念：**
- MCP 架构：Claude Code（MCP Client）↔ JSON-RPC 2.0 ↔ MCP Server（暴露 Tools/Resources/Prompts）
- 三种传输方式：Stdio（本地进程，零配置）、HTTP（远程服务，支持 TLS）、SSE（流式推送）
- MCP 生态规模：9700 万月度 SDK 下载量，10000+ 公开 MCP 服务器，75+ 官方 Claude 连接器
- 与 LSP 的关系：MCP 复用了 Language Server Protocol 的消息流思想，服务的是 AI Agent 而非代码编辑器

**对 news1 项目的启发：**
- news1 的 Research Hub 已经通过 Exa MCP 连接外部搜索，是 MCP 的实际应用
- 可以考虑接入 GitHub MCP（自动同步 Issues）、Notion MCP（读取产品需求）等，扩展 news1 的数据源
- 本地 SQLite 数据库可以通过 MCP Server 暴露给 Claude，让 Claude 直接查询 content.db

## 第26篇：18｜庖丁解牛：Tools 工具系统内核剖析

**核心观点：** Claude Code 的工具集是精心选择的原语（primitives）而非特化的高级能力——软件工程的五个原子操作（感知/搜索/修改/执行/获取）通过 LLM 推理和反馈循环产生涌现，1+1 远大于 2。

**关键概念：**
- 五个原子操作：感知（Read）、搜索（Grep/Glob）、修改（Edit/Write）、执行（Bash）、获取（WebFetch/WebSearch）
- 涌现公式：涌现能力 = 原语工具 × LLM 推理 × 反馈循环
- Bash 是图灵完备的逃逸舱：可以执行操作系统能做的一切，因此需要最严格的权限控制
- 工具风险等级：只读工具（无需授权）→ 文件修改（需确认）→ Bash 执行（始终需要授权）

**对 news1 项目的启发：**
- 理解工具风险等级有助于为 news1 的子代理配置最小权限：采集验证子代理只需 Read+Grep+Glob，发布子代理才需要 Bash
- 涌现原理解释了为什么 news1 的简单工具组合（requests + sqlite + claude_call）能完成复杂的新闻处理任务

## 第27篇：19｜无人值守：Headless 模式与 CI/CD 集成

**核心观点：** Headless 模式（`-p` 标志）让 Claude Code 从"有人值守"变成"无人值守"——输入变成一次性 prompt，输出变成 stdout 文本或 JSON，可以嵌入 CI/CD 流水线、pre-commit hook、定时任务；PR 审查周期从 36 小时缩短到 8 小时。

**关键概念：**
- `-p` 标志：print 模式，不打开交互界面，直接执行任务，输出到 stdout 后退出
- 三种输出格式：Text（默认，人类可读）、JSON（包含元数据，程序解析）、Stream-JSON（实时流式）
- 关键参数：`--allowedTools`（限制工具范围）、`--max-turns`（防止无限循环）、`--output-format json`
- 判断标准：任务需要人类在过程中做决策→交互模式；输入输出在启动前就能完全确定→Headless

**对 news1 项目的启发：**
- news1 的 daily_run.sh 已经是 Headless 模式的实践（launchd 定时触发，无人值守）
- 可以用 `--output-format json` 获取执行元数据（耗时、token 消耗），用于成本监控和性能调优
- GitHub Actions 可以集成 Claude Code Headless 模式，在 PR 创建时自动进行代码审查

## 第28篇：20｜有章可循：Rules 规则系统深度剖析

**核心观点：** Claude Code 有两种完全不同的"规则"：指令规则（.claude/rules/，告诉 Claude 该怎么做，是员工手册）和权限规则（settings.json，告诉 Claude 能做什么，是门禁系统）；两套规则协同运作才构成完整的行为约束体系。

**关键概念：**
- 指令规则两种加载模式：全局加载（无 paths 字段，会话启动时加载）和条件加载（有 paths 字段，匹配文件时才加载）
- 一旦加载不会卸载：paths 控制的是何时加载，不是何时生效
- 拆分判断：CLAUDE.md < 200 行不用拆；200-500 行考虑拆；> 500 行必须拆
- 权限规则：settings.json 中的 allowedTools/blockedTools/permissions，是客户端的行为约束

**对 news1 项目的启发：**
- news1 的 .claude/rules/ 目录已经实践了条件加载（design.md/code-style.md/testing.md 等）
- 可以为不同文件类型添加 paths 条件：`collectors/**/*.py` 加载采集规范，`tests/**/*.py` 加载测试规范
- 当 CLAUDE.md 超过 200 行时，应该把领域规范拆到 rules/，保持 CLAUDE.md 精简

## 第29篇：21｜登堂入室：通过 Agent SDK 掌控 Claude Code

**核心观点：** Agent SDK 把 Claude Code 的所有能力封装成可编程接口，从"使用者"变成"构建者"——CLI 是手动操作，Headless 是自动化脚本，SDK 是可编程集成；`query()` 函数是最简单的方式，`ClaudeSDKClient` 是有状态的会话管理。

**关键概念：**
- Agent SDK：不是新的模型 API，而是对 Claude Code 这个 Agent 系统的完整封装，包含完整的 Agentic Loop
- 两种使用方式：`query()`（无状态，一次性调用，适合轻量级用例）和 `ClaudeSDKClient`（有状态，多轮对话，适合复杂应用）
- 安装：Python `pip install claude-agent-sdk`，TypeScript `npm install @anthropic-ai/claude-agent-sdk`
- 异步迭代器：`query()` 返回异步迭代器，逐条接收 Agent 产生的消息（支持流式处理）

**对 news1 项目的启发：**
- news1 的 Research Hub 可以用 Agent SDK 重构，把 Claude API 调用升级为完整的 Agent 循环（自动读文件、搜索、分析）
- 可以用 SDK 构建一个"新闻质量评估 Agent"，自动读取 content.db 中的新闻，分析质量并生成报告

## 第30篇：22｜得心应手：Agent SDK 高级应用

**核心观点：** Agent SDK 的高级特性包括自定义工具（@tool 装饰器，本质是进程内 MCP Server）、Hooks 拦截（SDK 层面的安全控制）、四层权限管理、流式会话；通过这些能力可以构建生产级的自动化测试修复 Agent。

**关键概念：**
- 自定义工具：`@tool` 装饰器定义工具，SDK 工具直接在应用进程内运行（消除进程管理和 IPC 开销）
- 三层解耦调用链：Agent（推理）→ MCP Server（路由中枢）→ Tools（具体能力）
- description 的重要性：不是给人看的注释，而是给 AI 看的使用指南，决定 Agent 何时调用该工具
- 四层权限管理：全局权限 → 会话权限 → 工具权限 → 操作权限（逐层收紧）

**对 news1 项目的启发：**
- 可以用 `@tool` 装饰器把 news1 的核心功能（查询 content.db、触发采集、生成稿件）封装为 Agent 工具
- 自动化测试修复 Agent 的模式可以直接应用于 news1：测试失败 → Agent 分析原因 → 提出修复方案 → 人工确认 → 自动修复

## 第31篇：23｜化零为整：Plugins 插件打包与分发

**核心观点：** Plugins 把散落各处的 Commands、Skills、Agents、Hooks、MCP 配置打包成一个可安装、可升级、可分享的整体——把知识变成资产，把重复变成复用，把个人经验变成团队能力；`/plugin install @company/dev-toolkit` 一条命令完成团队环境配置。

**关键概念：**
- 插件目录结构：`.claude-plugin/plugin.json`（必需）+ `commands/`、`agents/`、`skills/`、`hooks/`、`.mcp.json`（可选）
- plugin.json：插件的身份证，类似 npm 的 package.json，定义 name/version/description/author
- 三层插件来源：官方市场（Anthropic 维护）、社区市场（开发者贡献）、企业市场（公司内部私有）
- 插件 vs 手动配置：手动配置导致 15 人有 15 种微妙不同的开发环境；插件把"约定"变成"约束"

**对 news1 项目的启发：**
- 可以把 news1 的 Skills + Agents + Hooks 打包成一个 `news1-toolkit` 插件，方便在多台设备或多人协作时快速配置
- 插件化也有助于版本管理：当 Skills 更新时，通过插件版本号追踪变更
