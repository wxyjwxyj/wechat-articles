# 极客时间《Claude Code 工程化实战》笔记 Part 2（第12-22篇）

## 第12篇：06｜去芜存菁：高噪声任务处理——测试运行器与日志分析器

**核心观点：** 信噪比决策框架——如果一个命令输出超过 50 行，且你只关心其中不到 10 行，就应该用子代理；子代理的价值是去执行高噪声任务，然后只把结论带回主对话，主对话 token 消耗可减少 58%。

**关键概念：**
- 信噪比框架：输出中真正需要的信息占总输出的比例，低于 20% 就该用子代理隔离
- 噪声隔离的真实代价：不只是多花 token，更关键是后续每轮对话都要携带噪声，稀释 Claude 注意力
- 子代理输出格式设计：不是学固定模板，而是掌握"只返回决策所需信息"的方法论
- 模型选择：探索型任务用 haiku（更快更便宜），复杂推理用 sonnet

**对 news1 项目的启发：**
- daily_run.sh 的各步骤输出大量日志，可以用子代理隔离，只把关键结果（成功/失败/条数）返回
- pytest 测试输出是典型高噪声场景，test-runner 子代理应该只返回"通过/失败数量 + 失败原因摘要"

## 第13篇：07｜百舸争流：多任务并行探索与流水线编排

**核心观点：** 并行探索（独立任务同时进行）和流水线编排（依赖任务顺序执行）是子代理的两大核心应用模式；判断标准是"独立→并行，依赖→流水线"；流水线中需要设计"交接契约"确保阶段间信息不丢失。

**关键概念：**
- 并行探索：多个子代理同时从不同角度分析，最后汇总综合报告（适合：新项目理解、多维度分析）
- 流水线编排：每个阶段只返回摘要，主对话始终保持清洁，可随时介入做决策
- 交接契约：阶段间传递的信息格式约定，确保下一阶段能正确消费上一阶段的输出
- 混合模式：部分独立（并行）+ 部分依赖（串行）的组合编排

**对 news1 项目的启发：**
- daily_run.sh 的 4 个采集脚本已经是并行模式（`&` + `wait`），符合"独立→并行"原则
- build_bundle.py 的去重→打标→翻译→生成是流水线模式，可以设计更明确的阶段交接契约
- 可以为每个采集源设计专门的探索子代理，并行分析各源的数据质量

## 第14篇：08｜群策群力：Agent Teams 多会话协作架构

**核心观点：** Agent Teams 解决的是"认知协作问题"——当任务之间不只是执行依赖，而是认知依赖时（多假设互相关联、不同视角互相推翻），需要 Teammates 直接交流、互相挑战，而非只向主对话汇报。

**关键概念：**
- Sub-Agents vs Agent Teams：前者是"任务委托"（子代理只向主对话汇报），后者是"团队协作"（Teammates 可互相发消息）
- 启用方式：settings.json 中设置 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS: "1"`（实验性功能）
- Team Lead + Teammates：Lead 协调工作、分配任务、综合结果；Teammates 各自独立工作并可直接通信
- 适用场景：多假设并行验证、锚定偏见消除、单视角误判纠正、级联故障根因分析

**对 news1 项目的启发：**
- 当采集数据质量出现异常时，可以用 Agent Teams 让多个侦探子代理（数据库侦探、网络侦探、API侦探）并行调查并互相验证
- 目前 news1 的问题规模还不需要 Agent Teams，但了解这个模式有助于未来扩展

## 第15篇：加餐｜子代理专题总结 & 春节练习

**核心观点：** 子代理真正解决的不是"能力问题"，而是"工程风险"——上下文污染、权限越界、高噪声干扰、阶段性分工和可控编排；两层能力体系：结构化分工（Sub-Agents）+ 协作型认知（Agent Teams）。

**关键概念：**
- 子代理四种模式：只读型（权限边界）→ 高噪声处理（信噪比）→ 并行探索（独立任务）→ 流水线编排（依赖任务）
- 权限变化原则：只有修复阶段的 Fixer 有写权限，其他所有子代理都是只读的
- 贯穿始终的方法论：痛点驱动设计 + 信噪比框架 + 交接契约 + 编排者决策
- 核心判断：面对任何工程场景，能判断是否需要子代理、需要什么样的子代理、如何组合它们

**对 news1 项目的启发：**
- news1 的完整流水线可以映射到子代理五阶段：并行采集（只读探索）→ 数据质量检查（高噪声处理）→ 去重打标（流水线）→ 生成稿件（流水线）→ 并行验证（只读+执行）
- 每个阶段的权限应该最小化，只有最终发布步骤才需要写权限

## 第16篇：09｜触类旁通：SKILL.md 结构与触发机制

**核心观点：** Skills 是"可被语义触发的能力包"，本质是按需加载的认知结构；它把能力的"语义定义权"交给模型——人不再编排具体执行路径，而是定义能力的边界与含义，模型根据 description 在运行时决定是否加载。

**关键概念：**
- Skills vs 被动文档：Skill 有触发条件、执行流程、输出规范、工具约束，文档只是静态文本
- Agent 生态四大支柱：Tools（能做什么）、SubAgents（谁来做）、Hooks（什么时候检查）、Skills（怎么做以及何时做）
- SKILL.md 结构：frontmatter（name/description/tools/model）+ 正文（执行步骤/输出规范）
- 企业本体论：Skills = 企业的 SOP 体系，把专业流程封装为可语义调用的能力单元

**对 news1 项目的启发：**
- news1 已有 finish、frontend-design 等 Skills，可以继续把高频操作（如"分析今日新闻质量"、"生成周报"）封装为 Skills
- SKILL.md 的 description 是触发器，要写得足够精准，让 Claude 能在正确时机自动激活

## 第17篇：10｜令行禁止：任务型 Skills（斜杠命令 /Command）实战

**核心观点：** 任务型 Skill（disable-model-invocation: true）是把重复的对话模式变成可复用快捷方式的机制；Commands 已合并到 Skills，新建命令推荐使用 Skills 目录（支持辅助文件和完整 frontmatter）。

**关键概念：**
- 参考型 vs 任务型：参考型由 Claude 自动判断是否使用，任务型必须用户手动触发（/command）
- $ARGUMENTS 传参：`/fix-issue 123` 中的 `123` 通过 `$ARGUMENTS` 注入到 Skill 内容
- 动态上下文注入：`$CURRENT_FILE`、`$SELECTION`、`$GIT_DIFF` 等变量在执行时自动替换
- Skill 内置 Hooks：可以在 SKILL.md 中定义 hooks 字段，只在该 Skill 执行期间生效

**对 news1 项目的启发：**
- news1 的 finish skill 是任务型 Skill 的典型应用，可以继续把 `/daily-check`（检查今日采集状态）、`/publish`（触发发布流程）等封装为任务型 Skills
- 使用 $ARGUMENTS 可以让 `/fetch-source hackernews` 这样的命令更灵活

## 第18篇：11｜循序渐进：渐进式披露架构设计

**核心观点：** 渐进式披露（Progressive Disclosure）是解决 token 爆炸和信息噪声的架构模式——三层结构（目录页/章节/附录），大多数请求只需要部分资源，按需加载可将 token 消耗降低 90%+；本质是知识的投资回报率（Knowledge ROI）最大化。

**关键概念：**
- 三层架构：目录页（只有 description，扫描阶段读取）→ 章节（SKILL.md 正文，激活后加载）→ 附录（reference/templates/data，按需加载）
- 注意力稀释效应：上下文窗口虽大（200K tokens），但冗余信息会稀释模型对关键内容的注意力
- description 预算：所有 Skill 的 description 共享 15,000 字符总预算，超出会被截断
- 认知经济学：以最小 token 投入获得最高任务完成质量

**对 news1 项目的启发：**
- news1 的 CLAUDE.md 已经实践了渐进式披露（主文件 + rules/ 目录），可以进一步把 Skills 也设计为三层结构
- 当 CLAUDE.md 超过 200 行时，应该把领域规范拆到 rules/，只保留核心约定

## 第19篇：12｜珠联璧合：Skills 与 SubAgent 配合实战

**核心观点：** Skills 和 SubAgents 的组合有两个方向：SubAgent 包含 Skill（子代理预加载领域知识，成为领域专家）和 Skill 包含 SubAgent（通过 context: fork 让 Skill 在独立子代理中执行，保持主对话清洁）。

**关键概念：**
- 方向 A（SubAgent 包含 Skill）：通过 `skills:` 字段给子代理预加载知识，适合需要专业知识的长期维护型专家 Agent
- 方向 B（Skill 包含 SubAgent）：通过 `context: fork` 让 Skill 在隔离上下文中执行，适合研究型任务和重型生成
- 核心判断：这件事需要"另一个人"承担（Sub-Agent），还是只需要"多一本手册"（Skill）？
- agent 字段：Explore（只读探索）、Plan（规划）、general-purpose（通用，默认）

**对 news1 项目的启发：**
- 可以创建一个 `news-analyzer` 子代理，预加载 `news-quality` Skill（包含评分标准、过滤规则），让它专门负责内容质量评估
- 深度研究功能（Research Hub）可以用 `context: fork` 模式，在独立子代理中执行，不污染主对话

## 第20篇：13｜纲举目张：Skills 架构定位与高级能力

**核心观点：** Skills 处于 Claude Code 五层架构的"知识层"——工具层（能做什么）之上，智能体层（谁来做）之下；三级进化路径：SOP（标准操作程序）→ 专家系统（领域能力包）→ 组织智能（多 Skills + SubAgents 协同）。

**关键概念：**
- 五层架构：LLM（推理核心）→ Tool Layer（原子能力）→ Knowledge Layer（Skills/SOP）→ Agent Layer（SubAgents/Teams）→ Automation Layer（Hooks）→ Distribution Layer（Plugins）
- 四种设计模式：模板驱动（约束输出格式）、脚本增强（封装确定性逻辑）、分层知识（按频率分级加载）、工具隔离（权限是最重要的设计决策）
- 三级自测：大多数项目在第二级（专家系统）就能满足需求，不要过度设计
- Skills 的终极价值：把人类组织中积累的知识管理经验（SOP、专家系统、组织学习）技术化映射到 AI Agent 架构

**对 news1 项目的启发：**
- news1 的 Skills 目前在第一级（SOP），可以逐步升级到第二级（加入 reference/ 和 templates/）
- 四种设计模式中，模板驱动最适合 news1 的稿件生成场景（固定格式的公众号文章）

## 第21篇：14｜星火燎原：从 Claude Code 到行业开放标准

**核心观点：** Skills 从产品特性到行业开放标准只用了不到百天，原因在于三个本质属性：声明式（纯 Markdown，任何系统都能读）、自包含（一个文件夹，git clone 即安装）、知识本位（价值在知识本身而非格式）；标准的价值大于独占的价值。

**关键概念：**
- 27+ 平台原生支持：OpenAI Codex CLI、Google Antigravity、Microsoft VS Code、Cursor、Windsurf 等
- 声明式优势：不依赖 Python 运行时或 Node.js 环境，任何能读 Markdown 的系统都能理解
- 自包含优势：不依赖外部注册中心，复制文件夹到任何支持 Skills 的环境即可工作
- 平台经济学：当格式成为行业标准，每个人创建的内容都在增强你的生态（类比 USB 标准）

**对 news1 项目的启发：**
- news1 的 Skills 可以考虑开源分享，贡献到社区（如新闻采集、内容评分、公众号格式化等通用 Skills）
- 使用标准 Skills 格式意味着 news1 的工作流可以迁移到其他支持 Skills 的平台（如 OpenCode）

## 第22篇：加餐｜Skills 专题总结

**核心观点：** Skills 专题六讲的演进路径：从"会写 SKILL.md"到"理解 Skills 为什么是 AI 工程化的关键基础设施"；两层能力体系：知识工程（第9-11讲，让 AI 在正确时刻拥有正确知识）+ 系统集成（第12-14讲，让知识驱动整个智能体系统）。

**关键概念：**
- 知识工程三讲：知识的结构与触发（SKILL.md）→ 知识的操作化（任务型 Skills）→ 知识的经济学（渐进式披露）
- 系统集成三讲：知识 × 智能体（Skills + SubAgent）→ 知识在架构中的位置（五层架构）→ 知识的平台逃逸（开放标准）
- 四种模式综合运用：一个生产级 Skill 通常组合多种模式（模板驱动 + 脚本增强 + 分层知识 + 工具隔离）
- 核心暗线：从"人调度模型"走向"模型调度能力"

**对 news1 项目的启发：**
- news1 的 Skills 体系可以按两层能力体系来规划：先完善知识工程（把采集规则、评分标准、格式规范封装为 Skills），再做系统集成（让 Skills 驱动 SubAgents 协作）
- 参考 Java 现代化改造案例，可以为 news1 设计一套"新闻处理流水线 Skills"：采集规范 + 质量检查 + 格式化输出
