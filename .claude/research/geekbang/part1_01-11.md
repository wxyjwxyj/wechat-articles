# 极客时间《Claude Code 工程化实战》笔记 Part 1（第1-11篇）

---

## 第1篇：开篇词｜共生而非替代：极客和 AI 的共舞

**核心观点：** Claude Code 不只是 AI 编码工具，而是一种新的人机协作范式——开发者从"执行者"转变为"指挥者"，通过 Sub-Agents 和 Skills 组织多个智能体协作完成目标。

**关键概念：**
- Sub-Agents（子代理）：复杂任务拆解给多个专职角色，主代理是指挥官，子代理是专业兵种
- Skills（技能）：语义触发的能力包，AI 自动判断何时激活对应领域知识，而非等待用户手动触发
- 渐进式披露：按需加载知识，解决 LLM 上下文窗口限制

**对 news1 项目的启发：**
- 当前 daily_run.sh 已是"指挥者"模式，可进一步把各采集脚本设计为专职 Sub-Agents
- Skills 的"语义触发"思路可用于 build_bundle.py：根据内容类型自动选择不同的打标签策略

## 第2篇：直播回放｜不止 Skills，1 小时带你解锁 Claude Code 高级用法

**核心观点：** Claude Code 是由人类与 AI 共同参与的工程系统，四层架构（基础层、扩展层、集成层、平台层）构成完整能力图谱；企业 Agent 落地要学三大功课：经验复用（Commands/Skill）、过程治理（Hooks）、组织协作（Teams/Subagents）。

**关键概念：**
- 四层架构：基础层（Memory/Tools）→ 扩展层（Skills/Commands）→ 集成层（Hooks/MCP）→ 平台层（Plugins/SDK）
- 经验复用：高频任务沉淀为可复用单元（慢SQL排查、变更影响面分析、生成周报）
- 过程治理：执行链路关键点插入控制点，让检验、限制、通知和审计自动发生

**对 news1 项目的启发：**
- 日报生成流程可以用 Hooks 在关键节点插入质量检查（如内容长度、标签覆盖率）
- 高频任务（每日采集、打标签、生成稿件）已经是经验复用的实践，可进一步 Skill 化

## 第3篇：热点加餐｜从 OpenClaw 到 OpenCode：拆解爆款 Agent 的设计密码与工程范式

**核心观点：** 真正能落地的 Agent 产品都遵循相同的工程 DNA：Memory、Sub-agent、Skills、MCP、Hooks、Headless、长期记忆七个核心维度；不要追逐工具热点，要理解背后的工程本质。

**关键概念：**
- Agent 工程七维 DNA：Memory（记忆）、Sub-agent（子代理）、Skills（技能）、MCP（外部连接）、Hooks（事件驱动）、Headless（无人值守）、长期记忆
- 四阶段 Agent 诊断法：信息收集 → 假设生成 → 假设验证 → 修复
- 工具层 vs 系统层：会使用工具是入门，能建系统才是核心竞争力

**对 news1 项目的启发：**
- news1 已具备 Memory（CLAUDE.md）、Sub-agent（test-runner/code-reviewer）、Skills（finish/frontend-design）、Hooks（settings.json）、Headless（daily_run.sh）五个维度，可对照七维 DNA 补全 MCP 和长期记忆
- 采集失败排查可用四阶段诊断法：并行采集日志 → 生成假设 → 定向验证 → 修复

## 第4篇：热点加餐｜Harness 架构深度解析

**核心观点：** Claude Code 的核心是 Harness——包裹 LLM 的运行时基础设施，由 Tools、Context、Memory、Hooks、Permissions 五个组件构成；2026 年竞争差异化的重心已从 Model 转移到 Harness。

**关键概念：**
- Harness：Agent = Model + Harness，模型只负责推理决策，Harness 提供工具调度、上下文工程、安全执行、状态持久化
- Agentic Loop：感知→决策→行动→反馈的循环，复杂任务可能跑几十轮
- 上下文管理：对话历史接近 92% 时自动压缩，CLAUDE.md 和系统提示词在压缩后重新注入
- 内置工具哲学：少而精的原语（Read/Write/Edit/Bash/Grep），通过组合涌现出复杂能力

**对 news1 项目的启发：**
- news1 的 daily_run.sh 本质上是一个手写的 Harness，可以参考 Harness 五组件设计来完善：Tools（各采集脚本）、Context（CLAUDE.md）、Memory（content.db）、Hooks（待补充）、Permissions（待补充）
- 理解 Agentic Loop 有助于设计更健壮的重试和错误恢复机制

## 第5篇：真相大白：源码泄露事故带来的架构验证与工程启示

**核心观点：** 2026年3月 Claude Code 源码意外泄露（512,000行），验证了课程中推测的架构设计；大型代码库理解策略：/init 生成架构地图 → Git 上下文自动加载 → 文件读取去重 → 大结果落盘 → 自动压缩 → LSP 语义理解。

**关键概念：**
- Self-Healing Memory：先写数据再写索引，保证数据可用性优先于索引一致性
- 情绪感知正则：源码内置检测用户挫败感的正则表达式，触发安抚响应
- QueryEngine.ts：46KB 的引擎层，负责 LLM 调用/流式/缓存/编排
- 大项目检索策略：Git 上下文自动注入 + 文件读取去重 + 大结果落盘 + AutoCompact + LSP

**对 news1 项目的启发：**
- content.db 的 WAL 模式设计与 Self-Healing Memory 的"先写数据"原则一致，验证了当前设计的正确性
- 可以借鉴 LSP 语义理解思路，为 news1 的代码库建立更精准的符号索引

## 第6篇：热点加餐｜AI 工程范式 3.0：Harness Engineering 到底革新了什么？

**核心观点：** Harness Engineering 是 AI 工程范式 3.0，核心是"写规范的时代取代写代码的时代"——工程师的价值从编写代码转向设计更可控、可观测的 Agent Harness。

**关键概念：**
- AI 工程范式演进：1.0（Prompt Engineering）→ 2.0（RAG/Fine-tuning）→ 3.0（Harness Engineering）
- Harness Engineer 核心能力栈：上下文工程、工具设计、权限架构、可观测性
- 可控性 vs 自主性：Harness 设计的核心张力，需要在效率和安全之间找平衡

**对 news1 项目的启发：**
- news1 的 CLAUDE.md + rules/ 体系就是 Harness Engineering 的实践，可以继续完善可观测性（日志、指标）
- schedule.yaml + scheduler.py 是 Harness 的调度层，体现了"规范先行"的工程范式

## 第7篇：01｜登台远望：Claude Code 底层技术全景导览

**核心观点：** Claude Code 的真正身份是"可编程、可扩展、可组合的 AI Agent 框架"，而非简单的编码工具；其四层技术栈（Memory/Tools → Skills/Commands → Hooks/MCP → Plugins/SDK）构成完整的工程能力体系。

**关键概念：**
- Claude Code 四层技术栈：基础层（Memory + Tools）、扩展层（Skills + Commands）、集成层（Hooks + MCP）、平台层（Plugins + SDK）
- API Key + 中转：通过 ANTHROPIC_BASE_URL 环境变量支持自定义端点，国内可用
- CC Switch：社区工具，支持模型切换、MCP 管理、Skills 管理，本质是改写 settings.json
- OpenCode：最佳平替，支持 DeepSeek/Qwen/Kimi/GLM，规则系统（AGENTS.md）借鉴 Claude Code

**对 news1 项目的启发：**
- news1 已使用 cckeys.top 代理（ANTHROPIC_BASE_URL），与课程中的中转方案一致
- 四层技术栈可作为 news1 Agent 化改造的路线图：当前在基础层，可逐步向扩展层和集成层演进

## 第8篇：02｜过目不忘：Claude Code 记忆系统与 CLAUDE.md

**核心观点：** CLAUDE.md 是解决 AI "失忆症"的核心机制，本质是在 token 有限的情况下尽量少走弯路——每一行都应该经过成本收益分析：这行带来的 token 消耗期望，是否小于不写这行时发生错误的概率乘以纠错成本。

**关键概念：**
- 记忆层级：全局（~/.claude/CLAUDE.md）→ 项目（./CLAUDE.md）→ 本地（CLAUDE.local.md）→ 会话内
- CLAUDE.md 最佳实践：每次纠正 Claude 后说"更新你的 CLAUDE.md，这样你就不会再犯这个错误了"
- 多微服务场景：个人级（协作方式）+ 微服务级（服务本身）+ 架构级（跨服务契约）三层分离
- 渐进式更新：持续迭代 CLAUDE.md，直到能量化看到 Claude 出错率下降

**对 news1 项目的启发：**
- news1 的 CLAUDE.md + CLAUDE.local.md + rules/ 体系已经实践了多层记忆架构
- 可以更系统地维护 MEMORY.md，把每次踩坑的经验及时沉淀，减少重复犯错

## 第9篇：03｜分而治之：Sub-Agents 的核心概念与应用价值

**核心观点：** Sub-Agents 解决的核心问题是"上下文污染"——把高噪声的执行过程隔离出去，只让真正有价值的结论留在主对话中；子代理是 Claude Code 里唯一结构上允许"执行完即丢弃"的东西。

**关键概念：**
- 上下文污染：测试输出、日志、中间过程塞满主对话，导致 Claude 越用越"健忘"
- 子代理上下文机制："报文传输"模式而非"共享内存"——子代理只能看到主 Claude 显式传递的 prompt
- CLAUDE.md 继承：代理定义文件 > 项目 CLAUDE.md > 全局 CLAUDE.md（后加载覆盖先加载）
- 子代理不能动态创建 CLAUDE.md，只能读取已存在的文件

**对 news1 项目的启发：**
- news1 的 test-runner 和 code-reviewer 子代理已经实践了噪声隔离
- build_bundle.py 的翻译和打标签过程产生大量中间输出，可以考虑用子代理隔离，只把最终 bundle 结果返回主流程

## 第10篇：04｜量体裁衣：从 Sub-Agents 到 Multi-Agent 的工程指南

**核心观点：** 理解 AI 产品需要"架构式思维框架"——面对陌生产品，要问：它解决的核心工程问题是什么？选择的是单 Agent 还是多 Agent？是用上下文换智能，还是用架构换可控性？

**关键概念：**
- 四种 Agent 模式：单 Agent + 多 Tool → Handoff Flow（顺序交接）→ Router（路由分发）→ Multi-Agent（并行协作）
- 选型判断：任务是否需要"另一个人"承担（Sub-Agent），还是只需要"多一本手册"（Skill）
- 上下文工程：上下文窗口是稀缺资源，架构设计的核心是如何精准投放知识
- AI Agent 模式 = 人类社会分工的仿生学：前台/中台/后台 = Agent，子部门 = Sub-Agents，技能 = Skills

**对 news1 项目的启发：**
- news1 的 daily_run.sh 是 Handoff Flow 模式（采集→去重→打标→生成→发布），可以在关键节点引入 Router 判断是否需要人工介入
- 当前串行流水线可以在独立阶段引入并行 Sub-Agents 提速

## 第11篇：05｜明察秋毫：构建只读型安全审计子代理

**核心观点：** 权限边界是子代理最重要的工程价值——安全靠权限控制，不靠君子协定；只读型子代理通过限制工具集（去掉 Bash 和 Write）从工具层面彻底断掉修改能力，而非依赖 prompt 里的"请不要改"软约束。

**关键概念：**
- 最小权限原则：只给子代理完成任务所需的最小工具集（只读任务：Read + Grep + Glob，去掉 Bash）
- Bash 是万能工具：即使没有 Write/Edit，Bash 也可以通过 shell 命令修改文件，必须从工具列表中移除
- permissionMode: plan：让子代理在执行前先展示计划，但不是 frontmatter 的合法字段（需在 settings.json 配置）
- 痛点驱动设计：从真实工程痛点出发，理解"为什么需要这个子代理"，再设计职责边界

**对 news1 项目的启发：**
- news1 的 code-reviewer 子代理应该严格限制为 Read + Grep + Glob，去掉 Bash，防止"好心修复"引入新问题
- 采集脚本的只读验证（检查 content.db 数据质量）可以用只读子代理实现，不影响主流程
