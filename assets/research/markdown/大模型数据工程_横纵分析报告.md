# 大模型预训练数据工程：从清洗到配比的四代演进

> **副标题：OpenAI 的 Reddit 到 DCLM 的评分器——预训练数据质量工程的进化之路**

> **2026 年 5 月更新**：北京大学 DCAI 团队开源 DataFlow 框架——被称为"数据届的 PyTorch"，用自然语言描述数据需求即可自动完成清洗、去重、配比全流程。实验表明 10K 条高质量合成数据的效果超过 1M 条未经筛选的通用数据。同期，"On Data Engineering for Scaling LLM Terminal Capabilities" 首次系统论证了数据工程在 Scaling 中的杠杆效应。合成数据驱动的训练范式正在重塑"数据工程"的定义。

---

## 一、一句话定义

大模型预训练数据工程，是一套从互联网海量原始文本中**采集、清洗、去重、筛选、配比**，最终产出用于训练语言模型的高质量语料的系统性方法论。它回答的根本问题是：给你 240TB 的原始网页文本，你怎么把它变成让模型变聪明的"养料"？

> 🎯 **读完这篇你能**：理解从 CommonCrawl 原始网页到高质量训练语料的完整 pipeline，能判断在预算有限时优先提升数据质量还是数据数量。

## 二、技术背景：为什么数据工程成了大模型竞争的主战场？

### 2.0 前置知识

数据工程的底层概念（CommonCrawl 的格式、Token 的计算方式、清洗 Pipeline 的核心流程、数据配比的比喻等）已整理为独立的前置知识文档，供不熟悉背景的读者提前阅读：

> **📖 [大模型数据工程前置知识](./大模型数据工程_前置知识.md)** — 读完再读主报告

2022 年，DeepMind 的 Chinchilla 论文给行业泼了一盆冷水——大多数模型都是"欠训练"的，模型参数和训练数据量应该按 1:20 的比例匹配。由此推算：如果你有一个 70B 的模型，你至少需要 1.4T token 的数据来训练它。

但这还只是**数量**上的问题。更关键的问题藏在 DCLM 的一篇 2024 年论文里：DCLM-Baseline 数据集只用 2.6T token 训练的 7B 模型，在 MMLU 上达到了 64%，接近 Llama 3 8B 用 15T token 训出的 66%。**2.6T 对 15T——数据质量的差距相当于把数据量差了 6 倍**。

这就是数据工程的价值。

### 2.1 问题的规模

预训练数据的核心原料是 **CommonCrawl**——一个每月爬取约 50-80 TiB 网页文本的非营利项目。2025 年的 CommonCrawl 快照积累了超过 240T token 的原始数据。但这里面包含了大量的"垃圾"：

- 机器翻译的乱码文本
- SEO 垃圾站点的关键词堆砌
- 电商网站的产品目录
- 论坛的自动回复模板
- 广告弹窗的文字残留
- 跨网站的完全重复内容

一条典型的 CommonCrawl 数据流水线，需要从原始 WARC 格式中提取文本（约 1/10 的信息密度），经过语言检测、质量过滤、去重等多个环节，最终可能只剩 10%-30% 的原始量。

### 2.2 数据工程的三层架构

从工程角度看，预训练数据处理可以抽象为三层：

1. **采集层**：从 CommonCrawl、书籍、代码仓库、学术论文等渠道获取原始数据
2. **清洗层**：去噪、去重、语言过滤、安全过滤、PII 剥离
3. **配比层**：决定各数据源的混合比例，以及从大规模候选池中选取最优子集

前两层解决的是"把坏的扔掉"，第三层解决的是"把好的挑出来"——后者在 2024-2025 年成为最核心的研究方向。

## 三、纵向分析：从 Reddit 到 DCLM 的四代演进

### 3.1 史前时期（2018 以前）：有数据就行

在 Transformer 诞生初期（BERT、GPT-1），预训练数据工程几乎没有独立的存在感。BERT 用 Wikipedia + BookCorpus，GPT-1 用 BooksCorpus。逻辑很简单：找一堆不是明显乱码的文本，训完再说。

当时没人认真思考过"什么样的数据能让模型更好"——因为只要模型能跑起来就是胜利。这就像原始人发现火能烤熟肉，还没来得及研究什么火候烤什么肉最好吃。

### 3.2 第一代：质量代理信号（2019-2020）

#### 催化剂：GPT-2 的 Reddit Karma 实验

2019 年，OpenAI 在 GPT-2 论文里做了件很朴素但开创性的事：他们不再随便找个语料库，而是去 Reddit 上抓取获得至少 3 个赞的外链对应的网页。**Reddit 的点赞数就是质量的代理信号**——一个社区帮你完成了初步筛选。

这个"代理信号"的思路现在看来很简单，却是第一次有人认真思考"怎么从互联网上挑好数据"。GPT-2 仅用了 40GB 的数据，比今天的数据集小了几个数量级，但产生的论文在写作质量和连贯性上远超当时所有模型。

#### C4：第一个系统化清洗 Pipeline

同一年，Google 的 T5 论文带来了 **C4（Colossal Clean Crawled Corpus）**。C4 不只是个数据集，它定义了一套完整的清洗规则：

- 用 langdetect 做语言识别，只留英文
- 移除包含脏词的页面
- 移除包含占位符文本（"lorem ipsum"）的页面
- 移除包含过多标点符号的页面
- 移除过短或过长的文档
- 移除重复 3-gram 超过 50% 的页面

这些规则在今天看来理所当然，但在当时是革命性的——它第一次把"数据清洗"从"有就行"变成了"可以系统化定义和优化"的问题。C4 的规则集成为后续所有工作的基线，直到 2024 年，FineWeb 和 DCLM 仍以 C4 作为对比的对照组。

#### CCNet：分类器替代规则

就在 C4 发布后的一个月，Facebook AI 的 CCNet 来了。CCNet 做了一件比 C4 更激进的事：**用训练好的语言模型来替代人工规则**。

它在一个小型高质量语料（Wikipedia）上训练 n-gram 语言模型，然后用这个模型给 CommonCrawl 的每个文档算 perplexity——perplexity 越低，说明这个文档在风格上与 Wikipedia 越接近，质量就越高。

这是一个范式的转换：从"人定规则"到"模型判断"。CCNet 证明了语言模型本身可以成为数据筛选的工具，这个思路在 5 年后被 DCLM 和 FineWeb-Edu 推向极致。

#### GPT-3：多源混合的提出

2020 年的 GPT-3 把数据工程推到了一个新高度。不是因为 OpenAI 公开了多少细节——实际上他们什么都没公开——而是 GPT-3 论文里披露的数据配比（CommonCrawl 60%、WebText2 22%、Books 8%、Wikipedia 3%）成了后续所有工作的参考模板。

GPT-3 证明了"多样化数据源混合"的有效性。纯 Web 文本不够，需要加图书培养长文理解、加 Wikipedia 培养事实准确性、加代码培养逻辑推理。这个"多源混合"的思想直接影响了一年后的 LLaMA 和 The Pile。

**小结：第一代的核心特征是"用简单规则和代理信号来做初步筛选"。这时候的数据工程还没有独立的身份，只是模型训练流水线的一个附属环节。但关键的范式转换已经发生：从"有什么用什么"到"什么好才用什么"。**

---

### 3.3 第二代：开源运动与透明化（2020-2022）

这一时期最关键的推动力来自开源社区——EleutherAI、BigScience、Together Computer 等组织不相信只有大公司闭门才能做出好数据，他们用完全开源的方式复制并超越了工业界的数据工程方法。

#### The Pile：多样性与开源

2020 年末，EleutherAI 发布了 The Pile——825GB、22 个子集的开源预训练数据集。它不只是个数据集，更是一次宣言：预训练数据的构建应该透明、可复现、社区驱动。

The Pile 第一次系统地分析了数据多样性对模型的影响。论文中的 ablation 实验表明，混合多源数据可以系统性弥补单一源的不足——比如只从 Web 文本训练的模型在学术写作上表现极差，加入 PubMed 和 ArXiv 后显著改善。

但 The Pile 也有明显的工程粗糙：不同数据源的质量参差不齐，版权争议（Books3）为后来埋了雷。它更像一个"把所有东西扔进锅里"的早期尝试，为后续更精细化的工程实践铺了路。

#### ROOTS：多语言与数据治理

2022 年，BigScience 项目的 ROOTS 数据集把数据工程带入了一个新维度：多语言和数据伦理。

ROOTS 包含了 59 种语言的 1.6TB 数据，而且首次把**数据溯源和伦理审查**纳入了 pipeline。每个数据源都要追溯来源、记录许可状态、分析潜在偏差。这种"数据治理"思维在几年后越发重要——当版权诉讼成为 AI 行业的核心风险时，ROOTS 的实践方式看起来颇有先见之明。

不足之处也很明显：低资源语言的数据量严重不足，部分语言的语料只有几十 MB。多语言数据工程的不平衡问题至今仍未被解决，只是在 FineWeb-2（2024）中得到了部分改善。

#### LAION：模型评分进入图片领域

LAION-5B 是多模态领域的 C4。它从 CommonCrawl 提取 58.5 亿图片-文本对，用 CLIP 模型给每对匹配度打分，只保留高质量对。核心方法论与 CCNet 一脉相承——用训练好的模型来评定数据质量——但把应用场景从纯文本扩展到了图文多模态。

LAION 的 CLIP Score 过滤方法直接影响了后来 DCLM 的模型过滤方案，而它引发的版权争议（Stable Diffusion 基于 LAION-5B 训练后被艺术家起诉）也成为后续数据工程不可回避的问题。

#### LLaMA：公开数据也能打

2023 年 2 月，Meta 的 LLaMA 论文震动了整个行业。不是因为模型架构有什么创新，而是它证明了一个简单的命题：**只用公开数据，也能训练出比 GPT-3（175B）更强的模型**。LLaMA-13B 在多项基准上超越了 GPT-3。

LLaMA 的数据配方在论文里写得非常清楚：

| 数据源 | 占比 | 处理方式 |
|--------|------|---------|
| CommonCrawl | 67% | CCNet pipeline + linear classifier 过滤 |
| C4 | 15% | 保留原始过滤版本 |
| GitHub | 4.5% | 按编程语言分，过滤低星仓库 |
| Wikipedia | 4.5% | 20 种语言 |
| 书籍 | 4.5% | 古腾堡计划 + Books3 |
| ArXiv | 4.5% | 论文全文 |
| StackExchange | - | 问答对 |

LLaMA 的数据配比后来成为开源社区的标准参考。它验证了几条原则：CommonCrawl 虽然质量参差但量大管饱，高质量源（书籍、Wikipedia）应该过采样，代码数据虽然占比较小（当时）但贡献巨大。

**小结：第二代的核心特征是"透明化、开源化和社区协作"。数据工程第一次从"只有大公司能做的事"变成了"社区驱动的科研协作"。但这一时期的 pipeline 还比较粗放——主要是"清洗→训练"两步走，缺少对"到底哪些操作真正有用"的系统验证。这个问题要等到 2024 年才被 Dolma 和 FineWeb 解决。**

### 3.4 第三代：精耕细作的消融时代（2023-2024）

第三代的标志性变化是：数据工程从"我清洗了数据"进入到"我验证了清洗的每一步都有效"的阶段。核心方法论是**消融实验（ablation）**——把每个处理步骤独立出来，在受控实验中测试它对模型性能的贡献。

#### Dolma：处理步骤的中间态消融

2024 年 1 月，Allen AI 的 Dolma 论文做了一个之前没人做过的事：他们把数据处理的每个中间状态（原始→清洗→去重→质量过滤→最终输出）都保留下来，在每个中间点上训练一个 OLMo 模型，然后看哪个步骤真正提升了性能。

结果有些反直觉：某些过滤操作对下游性能的提升远不如预期。一些大家习以为常的清洗规则，在 Dolma 的消融实验下被证明效果有限。

Dolma 最大的贡献不是数据集本身（虽然 3T token 的规模也不小），而是它建立的**数据消融实验框架**——让数据工程从"你觉得好用"变成了"实验证明有效"。

#### RedPajama v2：40 维质量信号

2023 年 11 月，Together Computer 的 RedPajama v2 采取了不同的思路。他们没有在"怎么过滤"上预设答案，而是为每个文档计算 40 多个维度的质量信号（perplexity、重复率、URL 多样性、符号占比等），然后把这些信号连同文档一起发布。

下游用户可以根据自己的需求灵活组合过滤条件——比如做知识密集型任务时可以侧重教育质量，做创意写作时可以侧重风格多样性。这个"释放信号而非预设过滤"的设计思想后来被 FineWeb 借鉴，用于探索不同过滤策略的组合效果。

#### FineWeb：最透明的数据工程

2024 年 6 月，HuggingFace 的 FineWeb 论文让行业对数据工程的认识全面提升。FineWeb 不仅是一个 15T token 的预训练数据集，更是一套完整的、完全透明的数据工程方法论。

FineWeb 的消融实验揭示了一些此前长期被忽略的关键细节：

1. **MinHash 去重的阈值选择极其重要**。阈值设得太激进（去重率太高）会损失有效数据；太保守又保留冗余。最优阈值需要精细调节。

2. **URL 去重比文档级质量过滤更有效**。移除已知的低质量域名（如代码镜像站、垃圾站点等）比用分类器判断单个文档的质量更可靠——这跟大多数人的直觉相反。

3. **多项在之前工作中被认为重要的质量信号，实际效果有限**。比如 RedPajama v2 提供了 40+ 维信号，但 FineWeb 在 ablation 中发现其中大部分对模型性能没有显著影响。

4. **教育质量过滤的惊人效果**。FineWeb 用 Llama 3 70B 标注了 50 万个网页的教育质量（0-5 分），训练了一个 DistilBERT 分类器，然后用这个分类器从 15T token 中筛选出教育相关性高的 1.3T token（FineWeb-Edu）。结果：MMLU 提升 6.2%，ARC 提升 8.4%。

这就是 "FineWeb-Edu 效应"——一个专门训练的轻量分类器，通过一个维度的质量判定，产出了在推理任务上全面超越更大数据集的子集。

#### DCLM：数据筛选的标准化评估

与 FineWeb 几乎同期发布的 DCLM（DataComp for Language Models）走了一条互补的路。它的核心方法论是：**把数据筛选变成一个标准化的 benchmark**。

DCLM 提供了：
- 240T token 的统一候选池（CommonCrawl 预处理版）
- 固定的 OpenLM 训练框架（412M / 1.4B / 7B 三种规模）
- 53 个下游评估任务
- 两个赛道：Filtering Track（怎么筛数据）和 Mixing Track（怎么混合数据）

参赛者只需要把自己的数据处理策略跑一遍，然后在 DCLM 的标准化框架下训练和评估，就能公平对比不同数据策略的效果。

DCLM 的关键发现直接挑战了业界对数据规模的认知：**DCLM-Baseline 用 2.6T token 训练的 7B 模型，在 MMLU 上达到 64%（5-shot），跟 Llama 3 8B 用 15T token 训出的 66% 相差不远**。训练计算量只有后者的 1/6。

这个发现的意义在于——如果数据质量足够好，你不需要盲目扩大数据规模。2.6T 高质量 token 的效果可以接近 15T 一般质量 token。

**小结：第三代的核心特征是"用实验证明每一步的价值"。数据工程从"清洗→训练"升级为"假设→实验→验证→优化"的闭环。FineWeb 的 ablation 和 DCLM 的 benchmark 共同确立了数据工程的实验科学属性。**

### 3.5 第四代：模型辅助的质量再提升（2024-2025）

到了 2025 年，数据工程面临一个新局面：原始文本的"浅层质量"（语法正确、拼写无误、不是乱码）已经被前三代 pipeline 很好地解决了。问题是，质量还有没有提升空间？

答案是：有，而且空间还很大。关键在于一个反直觉的事实——**高质量和低质量数据的界限不是"好的英语 vs 破碎的英语"，而是"有用的信息密度 vs 无用的废话"**。

第四代的创新集中在这个方向：用更强的模型来筛选训练数据，甚至生成合成数据来补充训练。

#### DCLM 的 Model-Based Filtering

DCLM 论文中的一个关键实验是用一个小模型（OpenLM-1B）在候选数据上训练，然后用训练过程中的 loss 值来反向评估数据质量。核心假设是：如果某条数据在训练中 loss 下降快，说明它包含的"可学信息"多，是高质量数据。

这等于用模型自己来给数据打分——不是一个独立的质量分类器，而是让训练信号本身成为质量度量。DCLM 发现这种做法选出的子集比使用外部质量分类器效果更好。

这是一个递归的隐喻：**用模型来筛选训练更好的模型所需的数据。** 模型越强，筛选能力越强，产出的数据质量越高，又能训练出更强的模型。

#### FineWeb-Edu 的 LLM 蒸馏方法

FineWeb-Edu 的方法在产业链上更实用：用最强的模型（Llama 3 70B）来做标注者，把它的判断能力蒸馏到一个轻量分类器（DistilBERT）上。

具体流程：Llama 3 70B 评估 50 万个网页的教育质量（从 0 到 5 分），这些标注数据训练一个 DistilBERT 分类器，然后用这个分类器扫描 15T token 的全部数据。最终筛选出 1.3T token 的高教育质量子集。

把一个 70B 模型的能力"压缩"到一个轻量分类器中，让它可以处理 TB 级别的数据——这是模型辅助数据处理的实用化方案。

#### DCLM-Pool：社区驱动的竞赛平台

2025 年的 DCLM-Pool 把这一范式推到了极致。MLCommons 提供了一个 240T token 的标准候选池，任何人都可以提交自己的数据筛选策略，在标准化的框架下验证效果。

数据工程从"每个团队关起门来搞"变成了"社区竞赛"。你不需要再自己处理 TB 级的原始数据——直接在预处理好的池上做创新的筛选实验就好。这与 ImageNet 在计算机视觉中的地位异曲同工。

#### 合成数据：从 SFT 到预训练

合成数据的使用在 2024-2025 年经历了一个重要转向：从主要用于 SFT 和 RLHF，开始进入预训练阶段。

- **DeepSeek** 在 V3 的 14.8T token 中加入了大量的合成数学和代码数据（估计占 20-30%），用于增强推理能力
- **Phi-3（Microsoft）** 大量依赖合成教科书和练习数据，仅 3.8B 参数就达到了 7B 模型的性能——证明了高质量合成数据的 scaling law 优于低质量网络数据
- **Cosmopedia（HuggingFace）** 完全由 Mixtral 8x7B 合成生成，包含 300 亿 token 的教科书、博客、故事等

合成数据带来了一个"鸡生蛋蛋生蛋"的可能性：用模型生成数据 → 用这些数据训练更好的模型 → 更好的模型生成更高质量的数据 → 循环。

**小结：第四代的核心特征是"模型辅助数据——用更强的模型来筛选和生成训练数据"。从分类器到评分器到数据蒸馏到合成数据生成，模型本身已经成为数据工程流水线中最重要的工具。**

### 3.8 第五代萌芽：质量幻觉批判与数据价值重估（2025-2026）

2025-2026 年，数据工程进入了一个"自我反思"阶段。前三代的成功让人们普遍相信"高质量数据 > 低质量数据"，但第四代的模型辅助方法引发了一个被忽视的问题：**如果质量分类器本身有偏差，筛选出的"高质量"子集是不是也有偏差？**

**Apple 在 2025 年末发表的"The Data-Quality Illusion"** 直接挑战了 FineWeb-Edu 式分类器过滤的根基。核心发现：基于分类器的质量过滤在提升某些基准表现的同时，系统性降低了模型在分布外任务上的泛化能力。直白地说，**你在 FineWeb-Edu 上看到的 MMLU 提升，可能部分来自"过拟合高质量数据的分布"而非"更好的推理能力"**。

这个发现不是要否定 FineWeb-Edu 的价值——它的 MMLU 提升是真实的。但它揭示了一个更深层的问题：当前的质量评估体系（以教育性、事实性为核心）可能存在结构性盲区，过度过滤会损失数据的多样性和覆盖面。

**合成数据的模型崩溃问题也在 2025 年集中爆发。** ICML 2025 收录的多篇论文研究了合成数据训练导致的"模型崩溃"——当合成数据占比超过某个阈值，模型性能开始回退，表现为输出多样性下降、长尾知识丢失。北京通用人工智能研究院提出了一种无需迭代训练的合成方案，声称可以避免崩溃。这个问题现在还远没有解决，但已成为行业共识：**合成数据有用，但不能无限制地用。**

**NVIDIA Nemotron-CC（2025）** 代表了另一种路线——不追求"极致质量"，而是在 CommonCrawl 上构建了一套工业级的标准化数据处理 pipeline（NeMo Curator），产出万亿 token 的高质量数据集。它证明了"可复现的标准化流程"比"每个团队自己调参"更有效。

**FineWeb-HQ（2025-2026）** 和 **FLUX（2026.03）** 进一步推进了数据价值评估的方法论。FineWeb-HQ 把 FineWeb 数据进一步提炼为更小的"超高质量"子集。FLUX 提出了"数据值得训练"（Data Worth Training On）的评估框架——不再是单个文档打分，而是评估一批数据对模型训练的增量贡献。

**小结：第五代的核心主题是"反思和收敛"。行业在经历了 2022-2024 年的数据规模爆炸后，开始系统地质疑"质量过滤是不是做得太过了？""合成数据的天花板在哪里？""什么样的数据评估才是真正有用的？"这些问题在 2026 年还没有确定答案，但它们定义了数据工程的下一个阶段。**

### 3.6 去重技术的演进：一个独立的平行故事

去重是数据工程中最容易被低估但实际效果最大的环节之一。从 2019 年到 2025 年，去重技术保持了独立的迭代路线：

| 阶段 | 时间 | 方法 | 代表工作 |
|------|------|------|---------|
| 精确去重 | 2019-2020 | URL 去重、MD5 哈希去重 | C4、CCNet |
| 近似去重 | 2020-2023 | MinHash + LSH | C4、LLaMA、RedPajama |
| 语义去重 | 2023-2024 | 嵌入空间余弦相似度 | SemDeDup (Meta) |
| SSL 去重 | 2024-2025 | 自监督学习特征 | 探索中 |

去重技术的发展反映了一个核心洞察：**冗余数据不仅浪费计算，更可能损害模型泛化**——重复暴露在高度相似的文本下会让模型倾向于记忆而非理解。

FineWeb 的 ablation 揭示了一个关键细节：MinHash 去重的阈值不是越大越好。过于激进地去重（阈值设得太高）会损失大量有效数据，反而降低下游性能。最优阈值取决于具体的数据集和目标任务。

SemDeDup 的发现则更令人吃惊——在 LAION 子集上，移除 50% 的"语义重复"数据后，模型性能几乎不变，训练时间减半。这揭示了一个事实：数据集中存在大量"内容不同但语义冗余"的文档，它们对训练的边际贡献接近于零。

### 3.7 配比优化：从人工到自动化

数据配比的历史可以看作一条从"手调"到"自动"的进化路径：

#### 手调时期（2020-2023）
GPT-3：CommonCrawl 60%、WebText2 22%、Books 16%、Wikipedia 3%
LLaMA：CommonCrawl 67%、C4 15%、GitHub 4.5%、Wikipedia 4.5%、Books 4.5%、ArXiv 4.5%

这些配比都是靠经验和直觉调出来的。Chinchilla 给出了"总量应该多大"的指导，但没回答"各来源怎么分"的问题。

#### 自动配比（2023-2024）
**DoReMi（2023）** 首次实现了数据配比的自动化。核心创新是用 Group DRO（分布鲁棒优化）训练一个小代理模型，自动学习各数据域的最优权重。损失下降慢的领域获得更多权重。在 The Pile 上平均提升下游准确率 6.5 个百分点，训练步数减半。

**Data Mixing Laws（2024）** 建立了混合数据配比的数学形式——不同数据源间存在复杂的交互效应，不是简单的加性关系。通过小规模实验可以预测大规模混合的性能。

**RegMix（2024）** 用回归模型学习配比与性能的映射关系，在小模型上跑大量不同配比的实验，训练回归模型预测最佳配比。

这些工作的方法论共性：**用小模型做代理实验，把结果外推到大模型**。跟 DCLM 的"用小模型评估数据质量"一脉相承。

## 四、横向分析：各家的数据策略对比

> **场景判断：该领域竞品充分（场景 C）。** 各家 AI 公司在数据策略上既有共识（代码占提升、合成数据、去重心法），也有明显分化（数据规模偏好、多语言策略、透明度）。以下按 skill 框架的梯队划分逐一展开，选取最具代表性的 6 家做对比表。

### 4.1 前沿梯队（OpenAI / Anthropic / Google DeepMind）

skill 框架将这三家定义为"前沿梯队"——它们掌握着最强的模型能力，在数据工程上投入最大、走得最前沿，但也是公开信息最少的三家。

#### OpenAI：最早但最不透明

OpenAI 是数据工程的先驱——GPT-2 的 Reddit karma 过滤、GPT-3 的多源混合都是开创性工作。但 GPT-4 之后 OpenAI 变得极其不透明，数据配比、清洗方法、过滤策略全部没有公开。

据第三方估计，GPT-4 的训练数据约 13T token，来源基本沿用了 GPT-3 的框架：CommonCrawl + WebText + 书籍 + 代码 + Wikipedia。合成数据有使用但比例未知。

值得注意的变化是，截至 2024 年，约 45% 的 C4 数据集因为版权诉讼和网站 paywall 而无法使用。OpenAI 因此与多家新闻机构签订了数据许可协议，但这部分数据的具体规模和用法同样没有公开。

#### Anthropic：合成数据是核心武器

Anthropic 的数据策略跟其他公司有一个根本性的不同——它在预训练方面的公开信息最少，却最重视合成数据。

Anthropic 的 Constitutional AI 核心是一个**大规模合成数据生成流程**：模型生成回答 → 自我批评 → 根据宪法原则修订 → 形成训练数据。这与"从网上爬数据再清洗"的传统路线完全不同——数据不是被采集的，而是被生成的。

2025 年后，Claude Code 也被用于生成合成代码训练数据。Anthropic 的 Clio 工具实现了用户对话数据的自动化匿名化处理，这成为另一个数据来源。

透明度中心文档提到，Anthropic 的数据策略强调"过滤的激进程度"——宁可砍掉大量网页文本，也要确保剩余数据的质量可控。这与 OpenAI 的"规模优先"形成鲜明对照。

#### Google DeepMind：全模态覆盖

Google 的数据策略跟它的业务版图一样大——PaLM 是纯文本，Gemini 则是文本 + 图像 + 音频 + 视频 + 代码的全模态数据。

Gemini 论文披露了数据处理的基本框架：启发式规则过滤 + 基于模型的分类器 + 安全过滤 + 全局文档级去重。但具体的配比和过滤细节没有公开。

Google 的优势在于拥有 YouTube、Google Books、Google Scholar、Google Search 索引等多条数据管道，数据来源的多样性和规模在业界无出其右。Gemma 作为开源模型的数据处理 pipeline 是相对透明的，采用了全局去重和严格的质量过滤。

### 4.2 落后梯队（Meta）

skill 将 Meta 放在"落后梯队"——不是因为它的模型差，而是因为它在数据策略上处于追赶位置。但 Meta 在**数据透明的公开程度上恰恰是业界第一**。

#### Meta（LLaMA）：数据配比优化的教科书

Meta 的 LLaMA 系列是数据配方最透明的工业级模型。从 LLaMA 1 到 LLaMA 3，Meta 展示了一条从"规模扩展"到"配比优化"的清晰路径：

**LLaMA 1（2023）**：1.0-1.4T token。CommonCrawl 占 67%，代码仅 4.5%。核心贡献是证明了"只用公开数据可以打闭源模型"。

**LLaMA 2（2023）**：2T token。配方与 LLaMA 1 类似但过滤质量提升。

**LLaMA 3（2024）**：15T+ token（405B 版本）。这是数据配比发生质变的一代：

| 数据类别 | LLaMA 2 占比 | LLaMA 3 占比 |
|---------|:----------:|:----------:|
| 通用数据（网页、书籍） | ~86% | ~50% |
| 代码 | ~4.5% | ~17% |
| 数学与推理 | - | ~25% |
| 多语言 | - | ~8% |

代码数据从 4.5% 增至 17%（增加近 4 倍），数学与推理数据专门开辟了 25% 的配额。直接结果：LLaMA 3 在代码生成和数学推理上的能力远超 LLaMA 2——这是数据配比优化的经典案例。

Meta 还把 Tokenizer 从 LLaMA 2 扩展到新增 27,744 个 token（48.5% 非英文、30.2% 代码），反映了对多语言和编程能力的方向性投入。

### 4.3 独立实验室

#### DeepSeek：精炼数据 + 合成推理

DeepSeek-V3 的 14.8T token 数据是"精选自然数据 + 大量合成数据"路线的代表：

- **语言/语义评估**：对文档做多维度质量评估，移除低质量内容
- **综合去重**：跨整个语料库去除重复或高度相似内容
- **数据再平衡（Remixing）**：通过补充代表性不足领域的内容来平衡数据集
- **合成数据**：加入大量合成数学和代码数据（估计占 20-30%）

DeepSeek 的一个重要特点是训练过程零不稳定——全 14.8T token 的预训练没有出现任何 loss spike，训练成本仅 557.6 万美元。这背后是数据质量和配比的精心设计。

DeepSeek-R1 在 V3 的基础上增加了推理导向的合成数据（长链推理轨迹），使用基于规则奖励的强化学习训练。这是"合成数据 + 强化学习"路线的典型代表。

#### xAI（Grok）

xAI 的核心差异化在于 X（Twitter）平台实时数据的集成。Grok-3 的训练数据截至 2025 年 2 月，使用约 1,500 人团队做数据策展、分类和标注，聚焦 STEM、代码、金融等领域。Grok-3/4 引入了合成数据生成，聚焦多步推理和代码任务。具体数据规模和配比未公开。

#### Mistral AI

Mistral 的策略强调效率优先，利用 SMoE 架构在较小激活参数量下获得高性能。多语言焦点明确——强调欧洲语言（英法意德西）。Magistral 系列开始使用 GRPO 生成的大规模合成数据。Codestral 使用专用代码和 STEM 数据。总体转向精选高质量数据路线。

### 4.4 国内主要玩家

中国公司的数据策略有一个共同特征：中英双语 + 更大规模。各家都把自己的模型做成双语甚至多语言，数据量也普遍高于同期的海外模型。

**Qwen（阿里巴巴）**：18T token，是已知规模最大的公开数据量之一。使用前代模型 Qwen2 生成合成数据用于增强。分阶段训练策略，强调代码和数学。

**智谱 AI（GLM）**：中英双语焦点，构建大规模图像-文本描述数据集的精细策展流水线。智能体导向训练数据（工具调用、长链执行）。支持达 1M token 的长上下文。

**MiniCPM（面壁智能）**：独树一帜的小模型路线。2.4B 模型用超过 1T token 训练，**数据-参数比超过 160:1**，远超 Chinchilla 最优的 20:1。核心方法论是"模型风洞"——先在 ~100M 规模的代理模型上测试超参数和数据策略，再做全量训练。

**Kimi K2（月之暗面）**：15.5T token，四个关键领域（网页、代码、数学、知识）。使用 MuonClip 优化器保证超大规模训练的稳定性。智能体任务使用真实世界数据 + 模拟环境克服数据稀缺。

**字节跳动**：充分利用抖音生态数据，构建四维拓扑网络，通过细粒度聚类在降采样的同时保持语义多样性。30,000+ 名词和 2,000+ 动词的分类系统嵌入文化特征。截至 2025 年 12 月，豆包日均 token 使用量超过 50 万亿。

**零一万物（Yi）**：3T token 的多语言语料库，中英双语焦点。Yi-34B 在同等规模开源模型中性能领先。

### 4.5 横向对比总览

| 维度 | OpenAI (GPT-4) | Google (Gemma 2) | Meta (LLaMA 3) | DeepSeek (V3) | Qwen 2.5 | Kimi K2 |
|------|:-----------:|:-------------:|:-------------:|:------------:|:--------:|:-------:|
| 发布时间 | 2023-03 | 2024-06 | 2024-04 | 2024-12 | 2024-09 | 2025-07 |
| 训练数据规模 | ~13T (估) | 6T (2B) / 2T (7B) | 15T+ | 14.8T | 18T | 15.5T |
| 代码占比 | 不详 | 高 (含代码) | **~17%** | **显著** | 高 | 显著 |
| 多语言 | 英文为主 | **原生多语言** | **8%+ (30+)** | 中英 | 中英为主 | 英文为主 |

### 4.6 核心趋势

从横向对比可以清晰看到几个共识性趋势：

1. **代码占比普遍大幅提升**。从早期模型的 4-5% 到当前的 17%+，代码数据被公认为提升推理能力的关键杠杆。这是个一石三鸟的操作：代码数据既训练了编程能力，又强化了逻辑推理，还给模型提供了结构化的长程依赖建模经验。

2. **合成数据从"可选"变成"标配"**。2023 年以前合成数据主要用于 SFT 和 RLHF，2024-2025 年已经进入预训练环节。DeepSeek 的推理链合成、Phi-3 的教科书合成、Cosmopedia 的完全合成数据集，都指向同一个方向。

3. **数据规模仍在增长但增速放缓**。头部模型从 2020 年（GPT-3）的 300B token 增长到 2024-2025 年的 15T+ token，五年增长了 50 倍。但 DCLM 的 2.6T token ≈ 15T token 的实验结果暗示，继续扩大规模可能已经不是最优策略。

4. **配比优化成为新的竞争焦点**。LLaMA 3 把代码占比增加了近 4 倍，数学和推理数据占比达到 25%，而多语言数据只占 8%——这些决策背后有明确的"期望能力导向"逻辑。你是想模型什么都会一点，还是在特定能力上做到极致？配比决定了一切。

## 五、横纵交汇洞察

### 5.1 历史的锁扣：为什么数据工程走到了今天？

把纵向演进和横向格局放在一起看，能发现几条清晰的因果链。

**第一条链：从"量"到"质"的范式转移**

纵向看，数据工程经历了从"WebText 的 40GB"到"C4 的 750GB"到"The Pile 的 825GB"到"LLaMA 的 1.4T"到"FineWeb 的 15T"到"DCLM 的 240T 候选池"——数据规模的膨胀曲线几乎是垂直的。

但转折点发生在 2024 年。当 DCLM 证明 2.6T 高质量 token 的效果接近 15T 一般质量 token，当 FineWeb-Edu 用 1.3T token 在推理任务上超越更大数据集，行业开始反思一个根本问题：**我们真的需要那么多数据吗？**

横向看，这个反思的落地速度在各家之间差异巨大。Meta 在 LLaMA 3 上已经用 17% 的代码占比 + 25% 的数学推理占比做出了选择，Qwen 仍然在走"大而全"的 18T 路线。两条路现在都能出好模型，但方向的分化已经开始了。

**第二条链：开源社区对透明度的倒逼**

OpenAI 的不透明（GPT-4 的数据策略至今是黑盒）反而催生了开源社区对透明数据工程的极致追求。从 EleutherAI 的 Pile 到 HuggingFace 的 FineWeb 再到 MLCommons 的 DCLM-Pool，每一条开源数据集的发布都在推高行业对数据工程透明度的期待。

横向看，这造成了有趣的分化：Anthropic 和 OpenAI 在闭门状态下走合成数据路线，Meta 和开源社区走透明化路线。**透明路线产出了更好的方法论（ablation 框架、benchmark 平台），但不透明路线在原材料（合成数据）上更激进。** 两条路线谁会在 3-5 年后胜出，取决于合成数据的天花板在哪里。

**第三条链：模型本身才是最好的数据工具**

最反直觉的洞察在于第四代的发现——用模型来筛选数据，比任何人工规则都有效。而且这个循环是可以自增强的：更强的模型 → 更精确的数据筛选 → 更好的训练数据 → 更强的模型。

这个递归逻辑指向一个结论：**数据工程的终点不是"更好的 pipeline"，而是"更好的模型成为 pipeline 的一部分"。** FineWeb-Edu 用 Llama 3 70B 做标注器，DCLM 用 OpenLM 做评分器——模型本身已经从数据工程的消费者变成了参与者和驱动者。

### 5.2 三个剧本

#### 剧本一（最可能）：配比优化的黄金时代

未来 1-2 年，行业会沿着 LLaMA 3 的路线加速——代码和推理数据占比持续提升，配比优化自动化（DoReMi 类方法工业化），合成数据稳定占据预训练数据的 20-30%。DCLM 类的 benchmark 成为数据工程的标配，每个模型研发团队都有一个专门的数据工程组来做 iterating 和验证。

这会是一条稳定爬坡的路线：模型的推理能力持续提升，但幅度不会像过去两年那么戏剧化。数据工程的竞争焦点从"谁的数据多多"变成"谁的配比更聪明"。

#### 剧本二（最危险）：合成数据的幻灭

合成数据的 scaling law 可能比预期的更早遇到天花板——模型生成的数据受限于生成模型自身的表达能力和偏差，多样性不足导致边际收益递减。如果合成数据超过 30-40% 后出现显著的性能回退，整个"用模型生成→用模型训练→用模型评测"的闭环可能形成"回音室效应"——模型之间互相学习，但都在一个受限的空间里。

如果这个剧本发生，数据工程会回到"找到更多更好的原始数据"的老路上，合成数据退回到仅用于 SFT 的定位。那些在合成数据上押注最多的公司（如 Anthropic）可能面临短期挫折。

#### 剧本三（最乐观）：数据工程成为验证范式的输出

模型的持续增强使"模型辅助数据"的循环达到临界点——最强模型几乎可以完全自动化数据筛选和生成 pipeline。数据工程不再是模型训练前的一个准备阶段，而是与训练并行运行、持续反馈的在线系统。

在这个剧本里，每个训练 checkpoint 都可以用来重新评估数据质量、动态调整配比、甚至生成缺失领域的新数据。数据工程和模型训练完全融合，不再有明确的边界。

剧本三的关键前提是"模型能力增长持续超越数据需求增长"——也就是模型变得足够强，强到可以近乎完美地筛选和生成训练自身所需的数据。从 DCLM 的 model-based filtering 来看，这个趋势的迹象已经出现了，但距离完全实现还有技术鸿沟要跨越。
