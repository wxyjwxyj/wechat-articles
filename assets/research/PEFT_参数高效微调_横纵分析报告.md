# PEFT / 参数高效微调横纵分析报告

> 从 Adapter 到 LoRA 到 QLoRA，从"加补丁"到"低秩分解"到"单卡 65B"，大模型微调如何在成本和质量之间找到第三条路。

---

## 一、一句话定义

PEFT（Parameter-Efficient Fine-Tuning，参数高效微调）是**在冻结预训练模型绝大部分参数的前提下，通过训练少量新增参数来完成下游任务适配的方法集合**。它的核心矛盾是：全量微调一个千亿参数模型的成本高到只有大公司能承受，但每个应用场景都需要专属适配——PEFT 让"一人一模型"从幻想变成了现实。

---

## 二、技术背景

### 核心矛盾：模型增长快于硬件增长

模型的规模在过去 6 年增长了 10000 倍（BERT 110M → 2026 年万亿参数模型），但单 GPU 显存只增长了 4-8 倍（16GB → 80GB-192GB）。这意味着全量微调的入场门槛越来越高：

- GPT-3 175B 全量微调需要数百块 GPU 运行数周，成本数十万美元
- 每个任务需要存储 350GB 的完整模型副本——部署 10 个任务就是 3.5TB
- 灾难性遗忘：在新任务上微调后，模型可能会忘记旧任务的能力

PEFT 的核心假设很简单：**预训练模型已经学到了足够丰富的知识，下游适配只需要对知识进行"微调"而非"重学"。** 如果"调整"这个动作本身可以用低维空间表示，那就不需要更新全部参数。

> 前置知识请参考独立文档 [《PEFT_前置知识》](PEFT_前置知识.md)，涵盖 LoRA、QLoRA、Adapter、Prefix Tuning 等基础概念的通俗讲解。

## 三、纵向分析：从 Adapter 到 LoRA 统治

### 3.1 前 LoRA 时代：Adapters 和 Prompt Tuning 的探索（2019-2021）

LoRA 出现之前，PEFT 领域有两条独立的技术线在发展。

**Adapter 系列（2019-2020）**：Neil Houlsby 和 Google 团队在 2019 年 2 月提出 Houlsby Adapter（arXiv:1902.00751）——在 Transformer 每一层的 self-attention 和 FFN 之后插入 bottleneck 适配器（降维→非线性→升维）。在 BERT base 上，只训练 3.6% 的参数就能达到全量微调 99.6% 的性能。

Adapter 的开创性在于它首次证明了"冻结主体、只训补丁"的可行性。但它有一个根本局限：推理时必须串行通过 adapter 模块，引入额外计算延迟。这对于延迟敏感的生产环境是个硬伤。

后续的 AdapterFusion（2020.05）和 AdapterDrop（2020.10）分别在多任务组合和动态效率上做了改进，但没有解决核心的"延迟问题"。

**Prefix/Prompt Tuning 系列（2021）** 走了另一条路——不在模型中插入新模块，而是在输入或 hidden state 前加"可学习的虚拟 token"。

Prefix Tuning（Li & Liang, 2021.01, arXiv:2101.00190, ACL 2021）在 Transformer 每一层的 hidden states 前拼接可学习向量，只有 0.1% 参数。Prompt Tuning（Lester et al., 2021.04, arXiv:2104.08691, EMNLP 2021）进一步简化——只在输入层加 soft prompt。后者的关键发现是：**模型越大，prompt tuning 越接近全量微调**——T5 XXL（11B）上完全匹配。

P-Tuning v2（Xiao Liu, 清华, 2021.10, arXiv:2110.07602, ACL 2022）把"深层 prompt"的思路推向成熟，在每一层都加入 prompt token，解决了 prompt tuning 在小模型和序列标注任务上的不足。

这个阶段的本质是：**三种路线都在探索同一个问题——"怎么在最少打扰原始模型的情况下完成适配"。** 但答案都不完美——Adapter 有延迟成本，Prompt Tuning 有场景局限，而即将登场的 LoRA 找到了一条两者兼顾的路。

### 3.2 LoRA 的诞生与统治（2021-2022）

**LoRA（Low-Rank Adaptation，2021.06, arXiv:2106.09685, ICLR 2022）** 由 Edward Hu 和 Microsoft 团队提出。它的核心洞察极其简洁：**模型微调时权重的变化量 ΔW 是低秩的**——不需要用一个大矩阵来表示"调整了多少"，两个小矩阵的乘积就足够了。

```
h = W₀x + BAx

W₀: 冻结的原始权重（4096×4096）
B:  小矩阵（4096×r）
A:  小矩阵（r×4096）
r:  秩，远小于 4096
```

用直观的方式理解：4096×4096 = 16,777,216 个参数，而 4096×16 + 16×4096 = 131,072 个参数——训练参数减少了 128 倍。

LoRA 在与当时两种主流方法的对比中展现出了压倒性优势：

| | Adapter | Prefix Tuning | LoRA |
|--|---------|--------------|------|
| 推理延迟 | 有（串行模块）| 无 | **无**（可合并）|
| 训练参数 | 3.6% | 0.1% | 0.01-1% |
| 效果 | 高 | 中（因任务而异）| 高 |
| 与量化兼容 | 有限 | 有限 | ✅ |

没有推理延迟是 LoRA 击败 Adapter 的关键——Adapters 的技术质量不差，但行业不会为"效果差不多但慢一截"的方案买单。**低秩分解不增加推理路径**——训练完后 BA 可以被合并回 W₀，推理时 W' = W₀ + BA 跟 W₀ 的维度完全一样。

到了 ICLR 2022 发表时，LoRA 已经被社区广泛采用，并开始扩散到 NLP 之外（Stable Diffusion LoRA, 2022.12，成为图像生成领域最流行的概念训练方法之一）。

### 3.3 QLoRA：PEFT 的"桌面化"时刻（2023）

**QLoRA（2023.05, arXiv:2305.14314, NeurIPS 2023）** 是 Tim Dettmers（bitsandbytes 作者）的作品。它把 PEFT 的普及推到了一个新的水平——不仅是"参数高效"，还是"硬件高效"。

QLoRA 没有发明新的 PEFT 方法，而是把三个已有技术组合在一起：

1. **NF4（4-bit NormalFloat）**——信息论上对正态分布权重最优的 4-bit 数据类型
2. **双重量化（Double Quantization）**——对量化常数再做一次量化
3. **分页优化器（Paged Optimizers）**——利用 CPU 内存处理显存峰值

结果：65B 模型在单块 48GB GPU 上微调 24 小时，达到 ChatGPT 99.3% 的性能。在此之前，微调 65B 模型需要 8 块 A100。

QLoRA 的核心影响不是技术上的（三个组件都是已有的），而是**经济上的**——它把"微调大模型"这件事的硬件门槛从"8 块 A100"降到了"1 块 A100"，对于个人开发者和中小企业来说，这是从"不可能"到"在预算内"的跨越。

### 3.4 百花齐放：DoRA、VeRA 与模型合并（2024-2026）

QLoRA 之后，PEFT 领域的创新从"新的 PEFT 方法"转向"更好的 LoRA 变体"和"模型合并"两个方向。

**DoRA（2024.02, arXiv:2402.09353, ICML 2024 Oral）** 来自 NVIDIA 团队，将预训练权重分解为**大小（magnitude）和方向（direction）**两个分量。方向用 LoRA 更新，大小用可学习向量更新。实验结果持续优于 LoRA，缩小了与全量微调的差距。

DoRA 的直觉很优雅：权重矩阵的本质作用不只是"幅度"（放缩输入），更是"方向"（决定输入在特征空间中的移动方向）。LoRA 同时改变两者，但方向信息可能比幅度信息更重要。分解后，更新可以更有针对性。

**VeRA（2023.10, arXiv:2310.11454, ICLR 2024）** 走向了极端轻量：所有层共享一对随机初始化的冻结低秩矩阵，只学习每层的缩放向量。参数量比 LoRA 减少约 100 倍。

**模型合并（Model Merging）** 是 2023-2026 年这个领域中一个有趣的方向——不训练模型，而是合并已有的微调参数。

- TIES-Merging（2023.06, arXiv:2306.01708, NeurIPS 2023）：三步法——修剪冲突量、解决符号冲突、只合并一致的参数
- DARE（2023.11, arXiv:2311.03099, ICML 2024）：发现 SFT 的权重更新极度冗余——随机丢弃 99% 的 delta 参数仍可恢复原能力
- 2025-2026 的 LoRA 合并：多个 LoRA adapter 的组合使用而非单一 adapter

### 3.5 Hugging Face PEFT 库：生态的整合（2023-2026）

Hugging Face PEFT 库（2022.11 发布）对 PEFT 领域的影响怎么强调都不过分。它统一了 40+ 种 PEFT 方法的编程接口——`LoraConfig` + `get_peft_model()` + `PeftModel`——让开发者用 5 行代码就能完成 LoRA 微调。

到 2026 年 4 月的 v0.19.1 版本，PEFT 库支持的方法从最初的 5-6 种增长到了 40+ 种：

- **LoRA 系列**：LoRA, QLoRA, DoRA, PiSSA, AdaLoRA, LoRA-FA, LoRA-XS, VeRA, OLoRA 等
- **高级变体**：RoAd（混合批处理）、ALoRA（选择性激活）、SHiRA（稀疏高秩）等
- **Prompt/Prefix**：Prompt Tuning, P-Tuning, Prefix Tuning

PEFT 库的版本演进本身就是 PEFT 领域创新的晴雨表——每几个月新增数种方法，说明这个方向仍然非常活跃。

## 四、横向分析：竞争图谱

### 4.1 场景判断

PEFT 领域竞品充分（场景 C），但竞争格局已经收敛——LoRA 是事实标准，其他方法在其子方向竞争。

### 4.2 PEFT 方法对比

| 方法 | 发布时间 | 训练参数 | 推理延迟 | 效果 | 适用场景 |
|------|:-------:|:-------:|:-------:|:----:|---------|
| LoRA | 2021.06 | 0.01-1% | **无** | 高 | **通用标准，首选** |
| QLoRA | 2023.05 | 0.01-1% | 无 | 高-极高 | **显存受限场景** |
| DoRA | 2024.02 | 0.01-1% | 无 | **极高** | 追求极致质量 |
| Adapter | 2019.02 | 3.6% | 有 | 高 | 历史参考 |
| Prefix Tuning | 2021.01 | 0.1% | 无 | 中 | 生成任务 |
| Prompt Tuning | 2021.04 | 0.01% | 无 | 中-高 | 大模型（模型越大越好）|
| P-Tuning v2 | 2021.10 | 0.1-3% | 无 | 高 | NLU 任务 |
| VeRA | 2023.10 | **0.0001%** | 无 | 中高 | 极致参数效率 |
| IA3 | 2022.05 | 0.01% | 无 | 中 | 最轻量 |

**关键结论：**

1. **LoRA 是 2026 年的通用标准**——原因不是它效果最好（DoRA 在某些任务上更好），而是它"效果好 + 无延迟 + 生态最全"的综合得分最高。
2. **QLoRA 是显存受限场景的唯一选择**——当你在消费级 GPU 上微调时，从 LoRA 换到 QLoRA 可以省 50-75% 显存。
3. **Adapter 已被 LoRA 取代**——推理延迟是硬伤。
4. **Prompt Tuning "模型越大越好"的发现**影响了后来的许多工作，但本身没有被广泛采用。

### 4.3 训练框架对比

| 框架 | 发布时间 | 核心优势 | 显存优化 | GitHub Stars |
|------|:-------:|---------|:-------:|:-----------:|
| **HF PEFT** | 2022.11 | 统一 API，40+ 方法 | 通用 | **21k** |
| **Unsloth** | 2023.11 | **2x 速度，70% 显存节省** | 极高 | **63k** |
| Axolotl | 2023.04 | 生产级端到端工作流 | 高 | 12k |
| LitGPT | 2023.05 | 可读性好，实现透明 | 高 | 13k |
| DeepSpeed | 2021 | 大规模分布式 | ZeRO 系列 | 37k |
| LoRAX | 2023 | 多 LoRA 并发推理 | 推理侧 | 4k |

**Unsloth** 是 2025-2026 年增长速度最快的 PEFT 训练框架（63k stars）。它的核心卖点是自定义 Triton 内核——通过 Padding Free 和 Packing 等技术，在 FFT/LoRA/QLoRA 上训练速度提升 2x，显存节省 70%。对于个人开发者而言，Unsloth 正在取代 Hugging Face PEFT 的"默认选择"地位。

**Hugging Face PEFT** 仍然是生态最全的选择——不是最快的，但支持的模型和方法最多（40+ 方法、几乎所有主流模型架构）。

**Axolotl** 是生产级的选择——单 YAML 配置全流程，支持多模态训练、强化学习微调（DPO/GRPO）、ND 并行等企业级特性。

### 4.4 LoRA 推理/服务化框架对比

| 框架 | 核心能力 | 多 Adapter 支持 | 最佳场景 |
|------|---------|:--------------:|---------|
| **vLLM** | 通用推理引擎 | 支持动态加载/卸载 | 生产级 LoRA 服务 |
| **LoRAX** | 专为 LoRA 设计 | **数千个** | 多租户 LoRA 服务 |
| **PEFT merge** | 简单合并 | 合并后不可热切换 | 单一任务部署 |

微调和推理的框架选择是分离的：你用 Unsloth/HF PEFT 训练 LoRA adapter，然后可以用 vLLM/LoRAX 提供服务。训练完的 LoRA 权重可以合并回原模型（`merge_and_unload()`），此时推理速度与原始模型完全相同。

## 五、横纵交汇洞察

### 5.1 LoRA 的"低秩假设"为什么成立？三个可能的原因

LoRA 有效的前提是"微调权重更新是低秩的"——这个假设在 2021 年提出时是被猜测的，到 2026 年我们有了三个可能的解释：

**1. 预训练模型已经学到了足够的特征空间**——下游任务只是在已有的特征空间中"微调"权重分配，不需要开辟全新的特征方向。因此更新量在特征空间中沿着几个主要方向进行，不需要高秩。

**2. 下游任务的数据量远小于预训练数据**——微调数据通常是几千到几万条，而预训练数据是数万亿 token。用少量数据做大的权重更新容易过拟合，低秩约束本身是一种正则化。

**3. DARE 论文的间接证据**——DARE 发现随机丢弃 99% 的微调 delta 参数，模型仍能恢复原能力。这说明微调产生的调整高度冗余，用一个低秩矩阵表示就足够了。

### 5.2 Adapter 被 LoRA 击败的真正原因

横向对比中，Adapter 在效果上不输 LoRA，但它"输"在了技术上不该被竞争的地方——**推理延迟**。

这是一个重要的行业规律：**在 LLM 领域，不能因为"训练时好"而牺牲"推理时"——推理调用次数是训练的上万倍，任何可感知的延迟增加都会被放大一万倍。**

LoRA 的 genius 不是它比 Adapter 效果更好——在某些任务上甚至略有不如——而是它在推理时零开销。Merge-and-load 是 LoRA 最被低估的设计决策。

### 5.3 QLoRA 的"组合式创新"成功说明了什么

QLoRA 没有发明任何全新的技术。NF4 是 bitsandbytes 的已有工作，双重量化是信息论的经典思路，分页优化器是操作系统的基础概念。QLoRA 的贡献在于**组合的时机和方式恰好命中了需求**。

它的成功说明了一个规律：**LLM 时代的创新很多时候不是"找到新答案"，而是"在正确的时间把已有的答案组装起来解决一个紧迫的问题"。**

全量微调 65B 的成本高到只有大公司能承受——这个"紧迫问题"在 2023 年达到了临界点。QLoRA 不是最佳的技术方案（如果是，QLoRA 之后的 DoRA 就不会有更好的效果），但它是第一个在"单卡 A100 微调 65B"这个用户故事上成立的方案。

### 5.4 PEFT 的"终局"在哪里

从 2019 到 2026，PEFT 的创新节奏从一个极端走向另一个：

```
Houlsby Adapter (2019):  3.6% 参数
Prefix Tuning (2021):    0.1% 参数
LoRA (2021):             0.01-1% 参数
VeRA (2023):             0.0001% 参数
```

创新在逐渐走向"参数最少"的极致——但参数最少不等于最好。LoRA 用 0.1% 的参数做到了 95-99% 的全量微调质量，VeRA 用 0.0001% 做到了 90-95%。

PEFT 的终极问题是：**如果适配只需要 0.0001% 的参数，那微调本身是不是在朝着"不需要"的方向演进？**

几个趋势指向了这个方向：
- **模型合并**（DARE/TIES）：不训练模型就能组合能力
- **Long CoT**（o1/R1）：模型在推理层面做"自适应"，可能减少对微调的依赖
- **Context Engineering**（2025-2026）：通过 prompt 设计而非模型修改来完成适配

### 5.5 三个未来剧本

**剧本一：LoRA 继续统治，QLoRA 普及到端侧（概率 60%）**

LoRA 保持 PEFT 事实标准地位。随着硬件能力增长（Blackwell B200 192GB 显存），更多场景可以直接跑 LoRA 而非 QLoRA。QLoRA 向端侧迁移——手机、笔记本上的个人化模型通过 QLoRA 完成适配。Unsloth 类优化框架进一步降低训练门槛，"点击即微调"成为可能。

**剧本二：模型合并和提示工程侵蚀 PEFT 领地（概率 25%）**

如果模型合并（DARE/TIES）的效果继续提升——用"选 adapter → 合并"代替"训练 adapter"——PEFT 的使用场景可能被侵蚀。结合 Context Engineering 的发展，"写 prompt + 选工具"可能替代大多数需要微调的场景。

但即使在这条剧本中，PEFT 也不会消失——它从"常规操作"退为"极致优化手段"。需要深度领域适配（医疗、法律）的场景仍然需要它。

**剧本三：全量微调成本大幅降低，PEFT 退居"快速实验"角色（概率 15%）**

如果推理成本的下降速度继续快于模型规模的增速（2023-2026 年推理成本下降了约 80%），加上更高效的分布式训练方案，未来可能全量微调 70B 模型的成本降低到跟今天 LoRA 微调一样。此时 PEFT 的"省钱"优势削弱，退居到"快速实验"和"原型验证"角色。

### 5.6 第一性原理追问

PEFT 的终极问题是：**训练全部参数是否真的有必要？**

LoRA 的答案是"没有必要"——权重更新是低秩的。
VeRA 的答案是"几乎不需要"——共享的随机矩阵 + 缩放向量就够了。
DARE 的答案是"99% 的更新都可以扔掉"。

这些答案指向一个共同的结论：**全量微调在做大量的"无效劳动"**——超过 99% 的参数量在做冗余调整。如果这个判断成立，那么全量微调的"成本合理性"本身就是需要被追问的。

但"微调是低秩的"不一定意味着"预训练也是低秩的"。**理解"学习"和"适配"的本质区别**，可能是理解大模型能力来源的关键——前者（预训练）确实需要大量参数来捕获世界知识的复杂性，后者（微调）只是在已有的知识结构中做细微调整。这两个过程的"信息密度"不同，所需要的参数规模也不同。
